我们这个 Agent 框架不要做成“聊天机器人框架”，而要做成：

可验证保险产品研究 Agent 工作台

核心不是模型会说什么，而是它能不能把一个结论拆成：

线索 → 官方来源 → 产品身份 → 资料组件 → RAG引用 → 计算/判断 → 报告 → 可复核轨迹

主框架建议：

LangGraph + ReAct节点 + FastAPI工具层 + RAG证据库 + 产品身份图谱 + Next.js轨迹台

一、总体架构

前端工作台

基于现有 E:\网站 调试轨迹台改。
展示任务输入、Agent节点、工具调用、证据来源、失败原因、最终报告。
不是普通 Chat UI，而是“研究过程可追溯 UI”。
Agent编排层

用 LangGraph 做主状态机。
每个节点只负责一个明确动作。
ReAct 只作为节点内部策略：需要搜索、观察、再决定下一步时使用。
工具层

官网规范读取工具。
本地 pdf_download_links.csv/jsonl 检索工具。
网络搜索工具。
官网页面/PDF验证工具。
中保协财产险自主注册产品查询工具。
RAG检索工具。
产品身份解析工具。
证据分级工具。
报告生成工具。
数据层

现有 RAG：Chroma + BM25 + SQLite。
新增证据图谱：产品别名、官方产品、主险、附加险、说明书、现金价值表、费率表、分红实现率。
证据必须保留 metadata：公司、产品名、来源URL、材料类型、下载时间、hash、来源等级。
财产险增加交叉验证源：中国保险行业协会“财产保险公司自主注册产品查询”
（https://cxcx.iachina.cn/）。该站点用于验证财产险产品注册/存在性与公司归属，不作为条款PDF下载源，
也不替代保险公司官网材料。

browser harness 使用边界：

中保协查询站如果需要登录、验证码或人工验证，允许由人手动完成登录与验证。
验证通过后，browser harness 可以在已授权浏览器会话内继续自动搜索、翻页、读取结果。
不得把账号密码、验证码、Cookie、Session Token 写入仓库或报告。
该能力记录为“人工解锁后的浏览器会话自动化”，不能描述成完全无人值守爬取。
评分与验收层

每次 Agent 运行都要打分。
分数不是“回答好不好听”，而是“证据是否闭环、引用是否官方、产品身份是否可信”。
二、核心流程

用户输入：

产品名、别名、公司、问题，或“帮我找某产品官方资料”。
Agent解析任务：

判断是找证据、做产品身份确认、做对比、查分红实现率，还是查隐藏条款。
本地证据检索：

先查我们已有的官网规范、spec、PDF清单、RAG metadata。
网络线索搜索：

找市场别名、渠道名、文章线索。
第三方只能当线索，不能直接当证据。
官网回溯验证：

回到保险公司官网或官方披露页验证。
找到条款、说明书、现金价值表、费率表等材料。
财产险产品交叉验证：

如果产品属于财产险，或公司为财产保险公司，额外调用中保协“财产保险公司自主注册产品查询”。
该查询结果用于确认产品名称、公司归属、注册信息或备案线索。
若中保协能查到产品，但官网材料未闭环，则只能进入“已注册/待官网证据闭环”状态。
若官网材料能闭环，但中保协查不到，要标记为“协会查询未命中”，进入人工复核。
产品身份建模：

把市场别名映射为官方组件组合。
例如：一个渠道名可能对应“终身寿主险 + 分红附加险 + 万能账户 + 现金价值表”。
RAG引用校验：

检查结论是否能引用到官方 chunk。
每个关键结论必须能追到原文、页码、URL、hash 或 metadata。
输出报告：

输出官方证据清单、证据等级、未闭环项、风险提示、可继续追踪的问题。
不输出 C端推荐、不做“最适合买哪个”。
三、可实施清单
阶段 1：框架边界确定
步骤：

定义 Agent 只做 B端产品研究，不做 C端投保建议。
定义证据等级：S1-S5。
定义正式报告只接受 S1-S3。
定义 S4监管/协会只验证，不下载、不商用引用。
定义 S5第三方只作线索。
定义财产险专项校验：中保协财产险自主注册产品查询属于 S4-注册核验源。
验收标准：

所有输出都能标注证据等级。
第三方内容不会进入正式引用。
报告没有“推荐购买”“收益承诺”“投保方案”。
财产险产品报告能区分“官网证据闭环”和“中保协注册核验命中”。
阶段 2：证据源注册表
步骤：

整理现有 data/各个保险公司PDF访问规范.md。
接入 insurance_harness/specs/*.json。
接入 cleaned/pdf_download_links.csv/jsonl。
新增中保协财产险自主注册产品查询源配置：入口URL、登录方式、查询字段、结果字段、人工登录说明。
为每条材料补齐公司、产品、材料类型、URL、来源等级。
标记哪些公司适合 requests，哪些需要 Playwright。
标记哪些来源需要 browser harness 人工登录/过验证后继续自动化。
验收标准：

输入公司名，能返回 canonical 官网入口。
输入产品名，能查到本地候选材料。
每条候选材料都有来源等级和材料类型。
输入财产险公司+产品名，能返回中保协查询状态：命中、未命中、需人工登录、查询失败。
阶段 3：产品身份解析
步骤：

建立“市场别名 → 官方产品名”的候选关系。
支持一个别名对应多个官方组件。
支持主险、附加险、万能账户、现金价值表、说明书等组件类型。
每个组件必须挂官方来源。
无官方来源的组件进入“待验证池”。
验收标准：

系统能解释“这个市场名为什么可能对应这些官方材料”。
组合产品不会被强行压成单一产品。
无官方闭环时，不进入正式报告结论。
阶段 4：官方证据验证 Agent
步骤：

先查本地材料库。
查不到再搜索网络线索。
搜到第三方线索后，回官网找正式材料。
验证 PDF 是否可访问、是否真 PDF、是否来自官方域名。
财产险产品额外走中保协自主注册产品查询；若需要登录/验证码，由人手动通过后再让 browser harness 自动完成剩余查询。
保存验证结果和失败原因。
验收标准：

输入一个产品名，能输出官方材料清单或明确失败原因。
PDF必须通过 URL、域名、HTTP状态、文件类型基础验证。
搜索失败时能说明失败在哪一步。
财产险产品必须说明：官网材料状态、中保协注册查询状态、两者是否一致。
browser harness 运行记录必须说明是否经过人工登录/人工验证。
阶段 5：RAG引用闭环
步骤：

将官方材料进入 RAG 前保留 metadata。
RAG chunk 必须带公司、产品、材料类型、URL、页码、hash。
Agent回答前先检索证据。
回答后检查引用编号是否真实存在。
数字、收益、现金价值、分红实现率必须有来源。
验收标准：

每个关键结论至少有一个官方引用。
引用能回到原始材料。
无依据时输出“不足以判断”，而不是编。
阶段 6：工作台接入
步骤：

现有调试台新增“证据验证 Agent”模式。
展示 LangGraph 节点。
展示 tool calls。
展示 source tier。
展示最终证据报告。
展示未闭环项和下一步建议。
验收标准：

B端产品岗能看懂 Agent 为什么得出结论。
失败不是黑盒，能看到卡在哪个工具或来源。
同一任务可以复跑和对比结果。
四、评分策略
总分 100 分。

官方证据闭环：30分

S1/S2官方来源充分，且财产险产品同时命中中保协注册查询：28-30
S1/S2官方来源充分：25-30
只有S3渠道材料：15-24
只有S4监管/协会/中保协注册核验，但没有官网材料：8-14
只有第三方线索：0-7
产品身份可信度：20分

别名、官方名、组件关系清楚：16-20
只有单一官方名但组件不全：8-15
无法确认身份：0-7
财产险产品如果“官网材料 + 中保协注册查询”都能对应同一公司和产品名，可在本项内取高分区间。
引用质量：20分

每个结论都有 chunk/页码/URL：16-20
主要结论有引用，部分缺失：8-15
引用弱或混乱：0-7
工具轨迹完整度：10分

每步工具调用、输入输出、失败原因完整：8-10
有主要轨迹但不完整：4-7
只有最终答案：0-3
合规安全：10分

无推荐、无承诺、无C端销售话术：8-10
个别表达偏销售但可修正：4-7
明显推荐/承诺收益：0-3
业务可用性：10分

B端产品岗可直接用于研究：8-10
需要人工补很多信息：4-7
只能当技术演示：0-3
五、阶段性通过线

60分以下：只能作为线索，不可进入报告。
60-79分：可进入内部研究草稿。
80-89分：可进入正式内部研究报告。
90分以上：可作为高可信样本沉淀进产品库。
最终方向
我们先不要急着做“会聊天的保险 Agent”。

第一阶段要做的是：

能找官方证据、能解释产品身份、能标注证据等级、能暴露工具轨迹的 Agent 工作台。

这才是后面做 IRR、现金价值、分红实现率、隐藏条款、产品对比表的地基。
